Revio Multi-Crew Runner

System documentation — architecture, crews, observability, prompt management, and deployment.

Revio Multi-Crew Runner is a Streamlit application that runs multi-agent AI workflows (crews) built with CrewAI. Every run is fully observable: spans are sent to Langfuse (and optionally Datadog LLMObs). Agent prompts are managed at runtime via Langfuse Prompt Management, so prompt versions can be promoted without redeploying.

Multi-crew UI
Streamlit tabs for Research and Fitness Training
Evaluation
Built-in Experiments tab + CLI runner for batch eval
Observability
Langfuse traces + Datadog LLMObs, both pluggable
Prompt Management
Prompts live in Langfuse with YAML fallback
Prompt Versioning
Every trace records prompt name, version, and label
Sessions
One Langfuse session per crew run

Architecture

crew_app.py (Streamlit)
    │
    ├─ _get_langfuse()           @st.cache_resource — one client per process
    ├─ _get_connectors()         ConnectorManager([LangfuseConnector, DatadogConnector])
    │
    └─ _run_crew(crew_name, inputs)
           │
           ├─ make_run_context(crew_name)   → RunContext (session_id, run_id, user_id …)
           ├─ EnrichedConnectorManager(connectors, context)
           │       injects RunContext into every span's metadata
           │       pushes session_id / user_id to each connector
           │       exposes crew_callbacks for CrewAI step/task hooks
           │
           └─ CREWS[crew_name]().run(inputs, obs)
                   │
                   ├─ PromptLoader.get(name, fallback=yaml_spec)
                   │       fetches from Langfuse (label="production")
                   │       merges {**yaml_fallback, **langfuse_config}
                   │       returns PromptResult(config, version, name, label)
                   │
                   ├─ crewai.Agent(**prompt.config)
                   ├─ crewai.Crew(agents, tasks, step_callback, task_callback)
                   │
                   └─ obs.span("crewai.research", "chain", metadata={…prompt versions…})
                           └─ kickoff_crew(crew, obs)
                                   step_callback → obs.span("agent.step",    "agent")
                                   task_callback → obs.span("task.complete", "span")

Adding a New Crew

  1. Create core/crews/<name>.py — subclass BaseCrew, implement crew_name and run().
  2. Create core/crews/<name>.yaml — define agents and tasks.
  3. Register in core/crews/__init__.py → CREWS dict.
  4. Add seed entries to seed_prompts.py → PROMPTS list.
  5. Add a UI tab in crew_app.py.

Span Hierarchy (per run)

crewai.research          [chain]   root — input + output + prompt versions in metadata
  crew.kickoff           [span]    wraps crew.kickoff(); captures stdout / stderr
    agent.step           [agent]   fires for every reasoning step (truncated to 2000 chars)
    task.complete        [span]    fires when each Task finishes

ConnectorManager

Fan-out coordinator in core/observability/__init__.py. Receives a list of BaseConnector instances; on every span() call it opens a span on all enabled connectors simultaneously and returns a MultiSpanHandle that broadcasts update() to all.

EnrichedConnectorManager (core/observability/context/enriched.py) wraps ConnectorManager. On construction it calls update_run_context(context) on each connector and merges RunContext.as_metadata() into every span automatically. It also exposes crew_callbacks — the get_crew_kwargs(obs) helper uses duck-typing to wire CrewAI callbacks if the addon is active.

Langfuse Connector

Uses Langfuse SDK v4 start_as_current_observation(). For each span it calls propagate_attributes(session_id, user_id, tags) inside the span context, propagating values to child spans via OpenTelemetry baggage.

Prompt Management Architecture

GitHub YAML  ──►  YAML fallback  ──►  deterministic default contract
Langfuse UI  ──►  runtime config ──►  rapidly-evolving AI behavior

YAML defines the starting-point defaults. Langfuse overrides them at runtime. Crews always work — even if Langfuse is unreachable — because the YAML fallback is always applied first.

PromptLoader

loader = PromptLoader()
prompt = loader.get(
    "researcher_agent",        # Langfuse prompt name
    fallback=yaml_agent_spec,  # dict: role, goal, backstory
    label="production",          # Langfuse rollout label
    cache_ttl=300,              # seconds (5-min local SDK cache)
)
agent = Agent(**prompt.config)

Resolution Logic

  1. If LANGFUSE_PUBLIC_KEY / LANGFUSE_SECRET_KEY are set, fetch from Langfuse.
  2. Merge: {**yaml_fallback, **langfuse_config} — Langfuse values override YAML.
  3. On any failure (network, prompt not found, missing env vars) → use YAML fallback silently.

Promoting a New Prompt Version

  1. Open Langfuse UI → Prompts → select prompt.
  2. Click + New version → edit role, goal, backstory in the Config block.
  3. Move the production label to the new version.
  4. Within ~5 minutes (cache TTL), all running crew instances pick up the new version.
  5. Each Langfuse trace records agent.<name>.prompt_version in metadata — compare versions in the Traces view.
Champion / Challenger: Promote the challenger to staging label. Pass label="staging" in PromptLoader.get() for an experiment run. Compare in Langfuse Experiments, then move the production label when satisfied.

Seeding Initial Prompts

Run once after first deploy or when adding a new crew:

py -3.12 seed_prompts.py

Creates version 1 of all 4 prompts in Langfuse, labeled production. Re-running creates new versions rather than overwriting.

Experiments — In-App Tab

  • Enter a Langfuse dataset name (default: crew-research-eval) and experiment prefix.
  • Runs ResearcherCrew against every dataset item with max_concurrency=1.
  • Each item gets its own EnrichedConnectorManager + fresh RunContext.
  • Results logged as Langfuse experiment runs — compare in Langfuse → Datasets → Experiments.
  • The form is locked while running (prevents double-submit).

Experiments — CLI Runner

py -3.12 run_experiment.py

Same logic as the UI tab. Useful for CI pipelines. Requires LANGFUSE_PUBLIC_KEY, LANGFUSE_SECRET_KEY, and OPENAI_API_KEY.

Streamlit Cloud Deployment

Add all required env vars in Streamlit Cloud app settings under Secrets (TOML format).

Pinned Dependency Versions

Streamlit Cloud uses uv which strictly enforces version bounds. The following pins prevent known conflicts:

PackagePinReason
crewai==1.14.4Exact version; newer may change APIs
chromadb==1.1.1crewai requires ~=1.1.0; >=1.2.0 conflicts
pydantic>=2.11.9,<2.13crewai needs >=2.11.9; chromadb pulls 2.11.7 transitively
openai>=2.30.0,<3.0.0crewai requires >=2.30.0; older cap caused failure
langfuse>=4.0.0,<5.0.0SDK v4 required; v3 no longer delivers traces to Langfuse Cloud

Hot-Reload Fix

crew_app.py purges all core.* entries from sys.modules before importing them. This prevents KeyError: 'core' on Streamlit Cloud's hot-reload cycles.

Datadog on Streamlit Cloud

Must use agentless_enabled=True and DD_TRACE_ENABLED=0 since there is no Datadog Agent sidecar. LLMObs is initialized before any OTel-using imports to avoid "Overriding of current TracerProvider" warnings.

Adding a New Observability Connector

  1. Create core/observability/<name>_connector.py.
  2. Subclass BaseConnector; implement enabled, span(), flush(), update_run_context().
  3. Add an instance to the list in _get_connectors() in crew_app.py.

No changes to crews, PromptLoader, or RunContext needed.

Key Design Decisions

DecisionRationale
ObsManager as a ProtocolCrews import nothing from core.observability; structural typing avoids circular imports on Streamlit hot-reload
TYPE_CHECKING guards in enriched.pyPrevents re-import of core.observability while crew_app.py is still loading it
One session per runEach crew run is a distinct Langfuse session; user_id still correlates the tab across runs
{**yaml_fallback, **langfuse_config} mergeYAML is a complete working default; Langfuse overrides selectively — missing Langfuse keys transparently use YAML
cache_ttl_seconds=300Langfuse SDK caches locally for 5 min; new label promotion is live within one cache window
@st.cache_resource for Langfuse clientSingle client per Streamlit worker process; avoids re-authenticating on every rerun
max_concurrency=1 in experimentsPrevents parallel CrewAI runs from interleaving stdout/stderr and hitting LLM rate limits
